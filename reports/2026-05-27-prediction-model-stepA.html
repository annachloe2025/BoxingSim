1. 背景

ユーザー提案を起点に立てた段階4プラン。 実物ボクサーは試合前のスカウト（事前情報）と試合中の読み更新（観測ベイズ）で 相手の次の挙動を予測しているが、現状の FSM は 即時観測しか持たず時間方向の期待値が存在しない。 この時間方向の空白を埋めるのが本タスクの動機。

軸本タスクの立ち位置
理念 「実物ボクサーが何を予測しているか」を機械化するのが動機。 「フェイント効いてほしい」を逆算しない（結論ありきの調整は禁止）
スコープ 既存の決定軸ごとに確率分布を持たせる。メタ層なし（予測する予測、を作らない）
分割 段階3 (α) Step A と同じ流儀で A（配管）／ B（接続）／ C（UI・調整）に三段分割

Step A の役割は、段階3 (α) Step A と同じく、後段 (B/C) で挙動を動かしたときの 差分が「Step B で動いた／Step C で動いた」と切り分けて読める 基準を作ること。

3. 予測の表現（プラン通り）

既存の決定軸ごとに「次フレーム何を選ぶか」の確率分布を保持。

軸値
attackTarget head, body
guardTarget head, body
state idle, attack, guard, move
attackKind jab, str, hvy, feint

Step A プライア既定値

attackTarget: { head: 50, body: 50 }
guardTarget: { head: 50, body: 50 }
state: { idle: 25, attack: 25, guard: 25, move: 25 }
attackKind: { jab: 40, str: 30, hvy: 15, feint: 15 }
modelDecay: 0.99 （実効ウィンドウ ~100F）
modelTrust: 0 （Step A は完全 no-op）

更新順序: 毎フレーム decay → 観測加算 → 正規化。 decay=0.99 は「実効的に直近 100F 程度の観測を効かせる」設計で、 試合長（数百 F）に対して半分弱の窓を覗く形になる。 ベイズ的にはディリクレ分布の擬似カウント更新と等価。

5. 予測モデル動作確認 — seed=100, 1 試合 176F

予測モデルの内部値（predictedOpp）は意思決定には影響しないが、 正しく更新されているかは top-1 一致率で確認できる。 「次フレームの実観測 = 予測分布の最頻クラス」と一致した割合を集計した。

軸 top-1 精度 random との差 備考
state 72.2% 25% → +47pt 4 値分類。プライアより大幅に学習が利いている
attackTarget 58.5% 50% → +8.5pt 2 値。微差だが 有意に偏り検出
guardTarget 100% (3/3) — サンプル少だが妥当な向き
attackKind 観測サンプル小 — Step B/C で集計強化

読み取り

  • 全 176 / 176 フレームで分布が変化 → 毎フレーム更新が機能
  • 初期分布: プライアそのまま（観測ゼロ時の振舞いも自然）
  • state 軸 +47pt は 「同じ状態が数フレーム続く」フレーム単位構造を予測モデルが拾えていることの裏付け （idle/attack/guard/move の遷移はスパースで、attack 状態などは複数フレーム継続）
  • attackTarget +8.5pt は head/body の選好偏りを プライア 50/50 から動かして検出できているシグナル

これらの数値は「予測モデルが動いていることの確認」であって、 Step B で意思決定に接続したとき試合がどう変わるかとは別の話。 Step B の効果測定では、別の指標（被弾率変化、フェイント効果など）で評価する。

6. 教訓・予防策

R6 verify.js ELEM stub バグの予防

段階3 (α) Step C で発覚した既存バグ（UI 要素追加時に verify.js 側 ELEM stub への登録漏れが sim 動作を壊す）の教訓に基づき、Step A 着手時に build() 内の 全 num() / lag() 呼び出しと ELEM stub を 必須突合した。

  • 今回は 未登録ゼロ（17 ID 全件登録済み）
  • Step A は UI スライダー追加なしで配管のみだが、Step C で UI を追加するときに この突合を 必ず行う

R5 Identity Check 失敗の予防

updatePrediction は Math.random を使わず、完全に確定的。 スナップショット参照と算術のみ。 Identity Check は MD5 比較で発覚するため、ここに乱数が混じった瞬間 PASS が崩れる。
