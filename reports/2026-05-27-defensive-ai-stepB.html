段階3 (α) Step B: guardScore 式の構造修正 + guardCommitment 有効化

⚠ 重要: このレポートの数値は部分的に無効

このレポートを作成した時点では、tools/verify.js の ELEM stub に gcP / gcE が未登録だったため、build() 内で parseFloat(undefined) が NaN になり || 0 でフォールバックされていた。

つまりこのレポートが示す数値は「guardScore 式変更だけの効果 （threat 依存比 3.67× → 2.29×）」であり、 guardCommitment は実は無効化されていた。

guardCommitment 真の効果と、その上に重ねた guardSwapStartup の効果、 および本バグ発覚と修正の経緯は 段階3 (α) Step C 完成 + 守備 AI 構造改善 総括 を参照。

守備 AI 構造改善 3 段プラン (A/B/C) のうち Step B 実装報告。 Step A で配管された guardCommitment を 0 → 0.5 で有効化し、 あわせて scoreActions の guard 系スコア式を組み替えて threat フラグ依存比を 3.7× → 2.3× に圧縮した。 挙動の構造変化を非対称 lag 7 段スキャンと性格 3 パターンで観測。

関連: 2026-05-27 Step A 配管 ／ 2026-05-26 perception-lag 中間 ／ 2026-05-26 段階3 完成 ／ プラン: ~/.claude/plans/ai-gleaming-hollerith.md ／ TASKS.md ／ LOG.md

1. 背景

perception-lag 中間 で抽出された守備 AI 構造的問題 3 点のうち、 Step B は #1（threatened フラグ依存） を構造的に解消することを担う。 Step A で配管と Identity Check（bit-identical PASS）まで済んでおり、本 Step では 既定値を no-op から実値に切り替えて挙動を動かす。

  • #1 threatened 依存: ガード score が threatened の有無で 0.10 / 1.20 と段差 → Step B で対象
  • #2 ガード切替コスト = 0F → Step C (guardSwapStartup) で対応予定
  • #3 一方的試合 → #1 と #2 の合成効果で観察すべきもの

5. 構造的解釈

良い変化（緑）

  • lag=0/0 で守備が「過剰機能」する形に増強。 block/clean = 2.44 で段階 2 Step C の方向に振れ戻り、 guardRangeBase 引き上げ + guardCommitment による同 target 強化が効いている
  • lag=3/5 以上で「守備の最低限」が機械的に成立。 threat boost を半減（0.80 → 0.45）したにもかかわらず block 件数は perception-lag baseline 比で逆増: lag=5/7 で baseline 52→80、lag=7/10 で 8→66。 threat 依存の構造低下は仕様通り
  • ガード切替頻度が適切に増えた。 非対称 lag=0/0 で Step A baseline 13.0 → Step B 18.2（+40%）。 殴り合い率も 9.0% → 10.6% に揃って上昇しており、 ガードの「使い分け」が増えた所作として読める

悪い変化・懸念（赤）

  • lag=0/0 の時間切れ 48/50 = 96% — プラン R1 基準（35% 未満）を大幅超。 無遅延条件では攻撃側が読まれすぎ、互いに当てに行けない泥沼が支配する
  • lag=3-7 で block/clean が 0.5 を割る（0.23 → 0.11 → 0.10 → 0.08）。 プラン目標を下回り、Step B 単独では構造的に解決できていない
  • 両攻撃型はほぼ無変化（df=0.2 では guardRangeBase 引き上げの効果が小さい — 設計通りだが、 「攻撃寄りの性格は守備改善の恩恵を受けない」事実は留保）

予想外の変化（黄）

  • 両守備型 lag=3/5 のガード切替 43.8 回/試合 — guardCommitment=0.5 は df=0.8 の guard score 拡大に対して 継続バイアスが弱い可能性。Step C の guardSwapStartup 導入で抑制されるか要観察
  • whiff「攻撃中」分類が lag 増加で激増 （lag=0/0 で 0% → lag=7/10 で 38%）。 ガード継続バイアスにより、相手が攻撃中でも自分は guard を続ける状況が生まれている
