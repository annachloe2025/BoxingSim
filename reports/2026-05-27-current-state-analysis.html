1. 実装済み機能の整理（FSM の現状）

状態（4基底 × グループ着色）

待機 / 始動 / 発生 / 硬直。状態グループは「待機・移動 / 攻撃 / 被弾(のけぞり) / ガード(ガード硬直含む)」の 4色でカード背景着色。stateLabel() は moveKey と stun から派生表示。

MOVES（19種、src/index.html:275-295）

系統内訳主なパラメータ
移動 5系統 前進/後退 ／ 上下ガード前進/後退 ／ ステップイン/バック ／ その場上下ガード 速 0.2〜0.6 ／ lock=1 ／ stopRec=1 ／ guard:true
攻撃 8種 ジャブ ▲/▼ ／ ストレート ▲/▼ startup=4 / dmg=5 ｜ startup=6 / dmg=12
　└ 強ストレート ▲/▼ 高威力・低速・ガードでもチップ startup=10 / dmg=22 / chipRatio=0.3 / blockstun=18
　└ フェイント ▲/▼ 振り出しだけ、当たらない startup=4 / active=0 / recovery=0 / dmg=0

target ∈ {head, body} を attack と guard 双方に付与。getGuardTarget(opp) === m.target でブロック判定。

主要パラメータ・機構

  • HP=100、KO、被弾＝のけぞり (stunType="hit")、ガード＝ガード硬直 (stunType="block")。被弾時は moveKey=null で動作中断
  • カウンター: 被弾時の相手フェーズで分類（始動 = ×1.5 / 発生 = 相打ち等倍 / 硬直 = パニッシュ等倍）。有利F は全種 ≤ +2（連続確定回避、グロッキー後送り）
  • スタミナ 100、攻撃と移動で消費・待機で +1.0/F 回復。閾値効果: damage 係数 0.8〜1.0 ／ startup ×1.0〜1.2 ／ 移動 speed ×0.85〜1.0（effectivePhaseLen）
  • AI: PERS 7軸 = 攻撃性 ／ 防御性 ／ 好む間合い ／ 我慢強さ ／ スタミナ意識 ／ presenceLag (既定 3F) ／ detailLag (既定 5F)。scoreActions で点数^2 確率選択、DECIDE_EVERY=3F
  • 観測: buildObs で self=完全 ／ opponent.x, hp = リアルタイム ／ ★フィールド (state, action, attackTarget, guardTarget, phaseRemaining, stamina) = detailLag 遅延 ／ ◯フィールド (isAttacking, isGuarding) = presenceLag 遅延

3. 直近で立ち上がった問題（perception-lag 中間 / phase3 完成版から）

A. 守備 AI の構造的問題 3 点（perception-lag 中間レポート由来、未解決）

① threatened フラグ依存

ガード score = (threatened ? 1.20 : 0.10) × (0.30 + df)。lag が入ると threatened が立つ前に被弾。 phase3 完成版で「inRange なら底上げ (0.30)」修正は入ったが、 「threatened に支配される」構造自体は残存。

② ガード切替コスト = 0F の前提崩れ

段階 2 時点は瞬時切替前提で守備が成立していた。lag 導入後も切替コストはゼロのまま。

③ 一方的試合

両者 lag=5F+ だと両者とも守備姿勢を取らず、先に当てた方の高速 KO。読み合いの圏が消失。

B. phase3 完成版で観測された残課題

  • カウンター頻発 550/50戦 ≒ 11件/試合（待つ局面が少なく殴り合い寄り）
  • body 率 49〜52% に固定。攻撃 target 選好が「相手 guardTarget の逆」のみで、相手 idle 中は guardTarget=null → 50/50 化
  • 非対称知覚（速い反応 vs 鈍い反応キャラ）未検証
  • フェイント連鎖は確率発生のみ。AI に「フェイント → 本命」評価式なし

5. 次の候補手の比較

候補動機（実物のどの仕組みか）trade-off
(α) 守備 AI 構造改善 実物ボクサーは「構え＝デフォルト」。threatened に依存せず常時ある程度ガードを選好すべき。 perception-lag レポートが (A) として推奨済み ○ lag 値ごとの再調整コスト解消・既存 3 問題を一気に潰せる
× 全体バランス再調整必要（攻めが通らなくなる方向にぶれる懸念）
既存パラメータの再チューニングのみで新 FSM 要素なし＝「機能網羅」より「足元の歪み修正」
(β) コーナー機構 + バックステップ回数制限 実物の D1。記憶 corner-loop の予想線「頭/腹 2 択 → バックステップ多発 → コーナー追い詰め」 ○ 空間（D1）という未着手領域に踏み込む新規性
× phase3 完成版で「バックステップ多発」は表面化していない （その場ガードで両択防御できてしまうため）。 記憶側でも「コーナー機構を入れる前に段階 3 でバランスを変える方が筋」と既に判断済み
(γ) ダウン / 立ち上がり / TKO (A3, A4) 試合の終わり方を増やす。実物として「HP 0 = 一発 KO のみ」は明らかにギャップ A2 (効き) を経由しないと「累積で崩れる」が定義できない＝ A2 → A3 の順序が筋。 今 A3 単独だと「HP 閾値で即ダウン」になり機械的に薄い
(δ) ラウンド制 + 判定 (E1, E2) 時間切れ試合の決着付け。「ジャブで削って判定」戦略を成立させる 単独では駆け引きの粒度に効かない。試合構造の外枠
(ε) 効きゲージ (A2) 蓄積で動作・反応・精度が崩れる A3 の前提条件として理にかなう。スタミナ・detailLag と動的に連動できれば 段階 3 のコア理念（動的知覚変化）に直結
(ζ) 非対称知覚 + フェイント連鎖評価式 個性軸の追加・既存 FSM を磨く 軽量・低リスク。phase3 完成版「次の方向」筆頭。F1 (技量) の先取り

推奨: (α) 守備 AI 構造改善

理由 3 点:

  1. perception-lag 中間レポートが明示的に「推奨」マーク付きで提案し、TASKS.md にも「将来候補」筆頭で残っている既知の宿題。 phase3 完成版の inRange 修正は対症療法であって構造改善ではない、と中間レポートが自ら指摘している
  2. 方法論「機械的仕組みを先に積む」に照らすと、現状の guardScore = inRange ? 0.30 : 0.05 + threatened ? +0.80 : 0 は「観測されたバランス問題を係数で押さえつけた状態」であって、 FSM の整合性として歪みが残っている。 新規機能（コーナー・ダウン）を上に積む前に足元を直す方が、後段の挙動の解釈可能性が上がる
  3. 同じ理由で fsm_gap_inventory.html の事実更新（17/5/0 → 実態反映）も同時にやっておくと、 次の選択肢の比較が機械的に行える

(β) コーナーは記憶 corner-loop 自身が「段階 3 でバランスを変える方が先」と判断済みで、誘発条件（バックステップ多発）も今は出ていない。 (γ) ダウンは A2 → A3 の順序が筋。 (δ) 判定は AI 構造を直した後に時間切れがどう振る舞うか観察してから入れる方が定義しやすい。
