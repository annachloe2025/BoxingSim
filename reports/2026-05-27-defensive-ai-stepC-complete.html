1. 背景

perception-lag 中間 で抽出された 守備 AI 構造的問題 3 点に対する 3 段プラン (A/B/C)。Step C は最後の段で、 ガード切替コスト = 0F 問題に対する処置を担う。

  • #1 threatened 依存 — Step B で guardScore 式を組み替え、 threat 依存比を 3.67× → 2.29× に圧縮（後述するバグの影響で当時の数値の解釈は要訂正）
  • #2 ガード切替コスト = 0F — Step C で対応（本レポート）
  • #3 一方的試合 — #1 と #2 の合成効果として観察

Step C の目的は、guardSwapStartup を 0 → 3F に引き上げ、 ガードを「上下打ち分けで意味を持つ姿勢」に変質させること。 副次的に Step C 実装中に既存バグを発見・修正したため、 Step C 観測は実質的に guardCommitment + guardSwapStartup の同時初実効化として扱う必要がある。

3. 重要な発見: 既存バグの発覚 — Step B の数値は部分的に無効

⚠ ELEM stub 不足による guardCommitment の長期無効化

Step C の実装で gsw を verify.js から動かそうとした際、 ELEM stub に gsw キーが存在しないことが発覚。 同時に gcP / gcE も未登録であることがわかった。

これは Step A → Step B を通じて存在していた既存バグ。 Step B 時点で PERS.{P,E}.guardCommitment を 0 → 0.5 に変更したが、 verify.js 側では build() 内で NaN → || 0 フォールバックされ、 実際は guardCommitment = 0 のまま観測されていた。

つまり Step B レポート の数値は 本来意図した「guardScore 式変更 + guardCommitment 有効化」の効果ではなく、 guardScore 式変更だけの効果（threat 依存比 3.67× → 2.29×）を示していた。 Step B レポート冒頭にも警告 panel を追加してある。

Step C の数値は (guardCommitment = 0.5) + (guardSwapStartup = 3) が初めて両方実効化された 結果として扱う必要がある。以降の比較は「Step B → Step C」と書いてあっても、 実態は「guardScore 式変更のみ → guardScore 式変更 + guardCommitment + guardSwapStartup」の差分である。

教訓: tools/verify.js の ELEM stub は src/index.html の DOM に対する mock であり、UI 要素を追加するたびに stub 側も同期する必要がある。 Identity Check は「式変更時の挙動一致」しか見ていないため、stub 不足のような 入力欠落型のバグは素通りする。

6. 構造的解釈

良い変化（緑）— guardCommitment が初めて実効化された結果

  • 両守備型での顕著な改善: ガ切替 43.8 → 27.5（lag=3/5、-37%）、34.4 → 17.7（lag=5/8、-49%）、 block/clean 0.70 → 0.77、時間切れ 38 → 18（lag=3/5）・19 → 9（lag=5/8）。 guardCommitment の継続バイアスが df=0.8 の guard score 拡大に対して 振動抑制方向に効いている
  • 両守備型の質的転換: Step B 時点の「過剰振動（ガ切替 43.8 / 試合）」から、 Step C で「機能的な使い分け（27.5 / 試合）」へ振る舞いの質が変わった。 ガード切替自体は半減した一方で block 数は逆増（351 → 450）しており、 「切替の量が減って、効いている切替の比率が上がった」読み
  • 非対称 lag=1/2 の改善: ガ切替 16.8 → 10.6（-37%）、時間切れ 29 → 18、P+E 勝 21 → 32。 lag=1/2 帯で「ガード継続バイアスが迷い時間を切り詰める」効果が出ている
  • R3 の懸念が機能側に振れた: Step B 時点で「guardCommitment が df=0.8 拡大に対して弱い」懸念があったが、 Step C で guardSwapStartup=3 と組み合わさることで継続バイアスが活き、 振動が抑制された

guardSwapStartup の効果は副次的（黄）

  • 主役は guardCommitment。改善の大部分は両守備型のガ切替低下と block/clean 上昇に集中しており、 これは継続バイアス起因と解釈するのが自然
  • gsw=3 単独の効果は chip 比率の微増程度: lag=0/0 の chip 率 15.3%（150/979）→ 18.3%（201/1097）に上昇。 切替に startup が乗ることでガードの「上下不一致」が増えて chip 経由の削りが微増した、というマイルドな効果
  • 非対称 lag>=3 帯では gsw=3 / gc=0.5 とも効果が小さい: 知覚遅延が大きい領域は「機械的にガードが成立しにくい」設計通りの構造で、 上下打ち分けの読み合いが成立する手前で被弾が決まってしまう

悪い変化・未解消（赤）

  • lag=0/0 時間切れ 48/50 → 45/50（90%）: わずかに改善はしたが、依然プラン R1 基準（35% 未満）を大幅超。 無遅延条件では攻撃側が読まれすぎて当てに行けず、 Step C 単独では構造的に解決不能と確定
  • 非対称 lag>=3 の block/clean < 0.5（0.22 / 0.13 / 0.10 / 0.06）: Step B から実質変化なし。知覚遅延が大きい帯で守備が機械的に成立しない問題は残存。 ただしこれは設計通りの構造であり、lag 既定値の選択側で吸収すべき

  1. lag=0/0 時間切れ 90%: Step C 単独では到達不能と確定。 lag を完全に 0 にする運用自体の見直し領域
