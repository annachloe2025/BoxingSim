段階4 (敵行動予測モデル) Step B: scoreActions の soft 化 + 観測 > 予測原則の導入

敵行動予測モデル 3 段プラン (A/B/C) の Step B 完了レポート。 Step A で配管・観測蓄積まで済んでいた予測分布を、scoreActions の oppGT / oppAtkT / threatened 分岐に 「観測 > 予測」原則で接続した（観測あり時は従来 hard 値、観測なし時のみ予測ベースでブレンド）。 MT (modelTrust) アブレーション 5 値 × 非対称 lag 7 段で観測し、当初プラン既定 MT=0.5 は プラン完了基準を満たさないと判定。観測ベースで既定値を MT=0.3 に下方修正した。 Step C の意思決定接続は完了とせず、Step C 前に観測指標追加と guardCommitment 感度分析を挟む方針。

関連: 2026-05-27 段階4 Step A（配管 + Identity Check） ／ 2026-05-27 段階3 (α) Step B（構成参考） ／ 2026-05-27 段階3 (α) Step C 完成総括 ／ プラン: ~/.claude/plans/ai-gleaming-hollerith.md ／ TASKS.md ／ LOG.md

1. 背景

Step A で「予測分布の構造体・更新ルール・観測 API への露出」までを配管し、 Identity Check (MD5 完全一致) と予測モデル動作確認 (state top-1 72.2% 等) を済ませた。 Step B の目的は、計算済みの予測分布を実際の AI 行動選択 (scoreActions) に反映させること。

  • 予測の使い所: scoreActions が見ている oppGT / oppAtkT / threatened の 観測ベースの hard 分岐。 観測ラグがあるシナリオでは数フレーム遅れて確定するため、その空白を予測で埋める
  • 守るべき原則: 観測 > 予測。観測が確定しているなら従来通り hard 値、 観測が空いているフレームでのみ予測でブレンドする
  • 分割: 段階3 (α) Step B と同じ流儀で、Identity Check 用フォールバック (MT=0) を残しつつ既定値で挙動を動かす

Step A レポートの「予測モデル動作確認」で、予測分布が 毎フレーム更新され、有意に偏りを検出している ことは確認済み。本 Step では、その分布を実際の意思決定に流したときに 試合構造が壊れずに変化する範囲 を観測ベースで定める。

2. 実装内容

2-1. scoreActions の soft 化（src/index.html）

oppGT (l.460 付近) / oppAtkT / threatened (l.408 付近) の hard 分岐を、観測 > 予測原則で 3 通りに分岐する soft 版に書き換え。

分岐条件挙動
観測あり oppGT !== null 等 従来通り hard 値（例: {0.3, 1.5}）。Step A までと bit-identical
観測なし + trust>0 oppGT === null && modelTrust > 0 予測でブレンド: (1 - trust) * 中立値 + trust * 予測ベース値
観測なし + trust=0 oppGT === null && modelTrust = 0 中立値（完全フォールバック）。Step A baseline と同じ振舞い
soft 接続式（概略）:
score = (oppGT !== null)
  ? hardValue(observation)
  : (1 - trust) * neutral + trust * predictedBased(predictedOpp)

2-2. PERS 既定値

  • PERS.P.modelTrust: 0 → 当初 0.5 → 観測後 0.3 に調整
  • PERS.E.modelTrust: 0 → 当初 0.5 → 観測後 0.3 に調整

当初プラン記載の既定値は MT=0.5 だったが、後述の verifier 観測結果を受けて 0.3 に下方修正。 理念整合: 「機械的観察に基づいて数値を直す」「結論ありきの調整は禁止」。

2-3. UI スライダー追加 新規

  • mtP / mtE: 0..100 表示で % として扱う。初期 value="30"
  • build() 内で PERS.{P,E}.modelTrust = num(mtP) / 100 等で注入

2-4. tools/verify.js 変更（R6 教訓どおり最初から登録）

  • ELEM stub に mtP / mtE を追加（初期 value="30"）
  • 環境変数 MT で modelTrust を切替可能に（アブレーション用）。 例: MT=0 node tools/verify.js、MT=0.7 node tools/verify.js
  • MT 未指定時のフォールバックは 0（UI 既定値 0.3 とは別管理）。これが Identity Check 互換性の鍵

3. Identity Check 結果（R5）

PASS（MT=0 で Step A baseline と意味的内容が完全一致）

MT=0 node tools/verify.js を Step A baseline と比較し、 意思決定および試合進行が bit-identical であることを確認。

  • scoreActions の soft 分岐は trust=0 で中立値に落ち、観測あり時も従来 hard 値なので 意思決定回路への寄与ゼロ が保証される
  • verify.js 側の MT 未指定フォールバックを 0 のまま維持 → 既存の検証スクリプトを書き換えなくても 自動的に baseline が出る
  • UI 既定値 (0.3) とは別管理。「ブラウザで開いたユーザーには予測が効く」「verify.js は baseline」が両立

段階3 (α) Step B / 段階4 Step A と同じ流儀で 「式を soft 化しても既定回路は baseline と同一」 を成立させた。この互換性があるからこそ、後段の MT アブレーションで「差はすべて MT に由来する」と 切り分けできる。

9. 構造的解釈の要点

予測モデルが「機械的に」機能している証拠

  • MT 上昇でガード切替頻度が線形に増加（lag=3/5 で 3.2→12.8 = 4 倍 at MT=1.0） → 「予測で先回りガード」の経路が確実に通っている
  • MT 上昇で殴り合い率は単調低下 → 先読みガード → 当たらない → 殴り合い減、の因果が成立
  • block/clean (lag=3/5) は MT 上昇で改善 (0.22→0.44) → 段階3 (α) で構造的に解決できなかった「lag≥3 の守備機能性」を予測モデルが補強

同時に発火した副作用

  • 両守備型ペアで予測の交互切替合戦: 両者が予測で先回りガードを切替合戦 → 攻撃通らず時間切れ。MT 1.0 で時間切れ 88%
  • これは guardCommitment 強化（Step C 想定）または 「予測の確信度ゲート」（Step C 後の改良候補）で対応すべき構造問題
  • attacker 側の予測利用 (predictedAtkT) は 攻撃 kind 分布が MT で不変 → 効いていない可能性。Step C 前に検証推奨

10. 次のステップ（Step C 前にやるべきこと）

verifier 推奨で、Step C 着手前に挟むべき観測・分析タスク 4 件:

#タスク目的
1 観測空白活用率の指標化（verify.js への新指標追加） 予測の効果を直接測定可能に。プラン完了基準 (3) が △ で止まっている問題の解消

3 低 lag (p1/d2) 時間切れ悪化の構造解明 観測がほぼ即時で得られるシナリオで予測を使う必要があるかの designer 検討。 「観測 > 予測」原則をさらに強化する余地

これらは Step C 内で扱うか、別タスクで処理するかは 要相談。 プラン上は次が Step C（modelPrior 設定 UI、modelDecay 調整、 性格との交互作用観察）だが、R3 の構造問題を Step C で同時に動かすと 切り分けが効かなくなる 懸念がある。
