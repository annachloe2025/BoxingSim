1. 背景 — なぜ判定システムを機械化するのか

ユーザー指摘:

「ガードが多くなる問題は、ガードが問題ではなく、ガードを崩す機構が無いことが問題」

実物ボクシングは「攻撃した側がポイントを取る」「ガードしているだけでは負ける」という 判定システムによる構造的圧力を持つ。 これを機械化することで 守備一辺倒 = ポイント負け の代償が自動的に生じる。

軸本タスクの立ち位置
理念 試合構造（ラウンド + 判定）を機械化するのが動機。 「ガードへの代償」は副作用として出現するもので、設計目的ではない
スコープ A=配管 / B=ラウンド境界・インターバル / C=採点・判定。 ガード崩し他機構（B スタミナペナルティ / C ガードクラッシュ）は 保留 or 別段階
関連既存実装 D 上下強制2択（攻め側が head/body を選び、守り側はどちらか一方しか守れない）は 既に実装済み相当。本段階は累積層を載せる位置づけ
段階4 との関係 段階4 Step C (modelPrior UI) は 保留中。 再開タイミングは段階5 完成後に判断

Step A の役割は、過去の Step A（段階3 (α) / 段階4）と同じく、 後段 (B/C) で挙動を動かしたときの差分が「Step B で動いた／Step C で動いた」と切り分けて読める基準を作ること。 今回は roundEnabled=false 既定で従来挙動を完全保存する形に倒している。

3. パラメータ既定値の根拠

定数の選定理由（プラン外で勝手に決めていない、設計根拠ベース）

定数既定値根拠
ROUND_LENGTH 300F 既存 MAX_FRAME=800 と同オーダー（3R × 300F = 900F）。 現状の試合長感覚に合う最短ラウンド長。UI から変更可
NUM_ROUNDS 3 多数決判定が成立する 奇数の最小値。 アマチュアボクシング 3R 相当
INTERVAL_LENGTH 60F スタミナ回復に充てるインターバル尺。 ラウンド長 300F の 1/5 で実物の 1R:60s / インターバル:60s に近い比率
STAMINA_INTERVAL_REGEN 2.0/F 60F × 2.0 = +120 で スタミナ上限 100 に対し実質完全回復。 「インターバルで完全に整える」設計
roundEnabled false Step A は 従来挙動の完全保存が目的のため既定 off。 Step B 着手時に既定 on へ切替

数値は実物ボクシングの構造（奇数ラウンド・インターバルでの回復）から逆算したもので、 「結論ありきの調整」ではない。性格パラメータと交互作用したときの実バランスは Step B/C 完成後に観測ベースで再評価する。

4. Identity Check 結果

PASS（完全 bit-identical）

編集前 vs 編集後 verify.js 出力（MT=0）の MD5 完全一致:

81eb68488e55b9902d1010da41d7dd71

  • roundEnabled=false 既定 → 新規 UI 入力欄が build() で無視される（実質 no-op）
  • 新規 state フィールド (round, roundFrame, matchPhase, roundScores, roundStats, decision, ended) は scoreActions / 接触判定 / ダメージ処理から参照されていない
  • stepSim 末尾の s.roundFrame++ は単なるカウンタで副作用なし （後段で >= ROUND_LENGTH 判定に使う準備）
  • verify.js 戻り値に新規 rounds キーが乗るが、Identity 比較対象の集計には使われていない
  • Math.random 不使用 → 確定的

段階3 (α) / 段階4 と同じく「配管を済ませてから式を変える」流儀。 Step B で endRound / stepInterval を有効化した瞬間に 初めて出力差分が出る — それが 「Step B によって生じた差」だと確定的に切り分けられる。

6. 教訓・予防策

R6 verify.js ELEM stub バグの予防（再掲・継続適用）

段階3 (α) Step C で発覚した既存バグの教訓に基づき、段階4 Step A から 「新規 UI 追加時は最初から ELEM stub に登録 + build() 突合チェック」を必須化している。 今回も 5 ID 全件登録済みで着手し、build() との突合まで実施した。

  • 新規 UI ID: 5 件 全件登録
  • build() 内 num() 呼び出しと突合 → 整合確認
  • Step B でラウンド表示用の UI を追加するときも、同じ突合を 必ず実施

R5 Identity Check 失敗の予防

新規 state フィールドの初期化と stepSim 末尾の roundFrame++ は Math.random 不使用で完全に確定的。 Identity Check は MD5 比較で発覚するため、ここに乱数が混じった瞬間 PASS が崩れる。 今回も roundEnabled=false 既定で従来経路を一切踏まない構造により PASS。
