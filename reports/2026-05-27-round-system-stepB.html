段階5 (ラウンドシステム + 判定システム) Step B: ラウンド境界処理 + インターバル + UI 進行表示

実物ボクシングの試合構造を機械化する 3 段プラン (A/B/C) の Step B 完了レポート。 Step A で配管した roundFrame / matchPhase / roundStats を駆動し、 roundEnabled=true 既定で ラウンド境界処理 + インターバル + UI 進行表示を有効化した。 部分リセット (partialReset) は HP・スタミナ・予測分布を持ち越し、フレーム状態を初期化。 50戦 × 複数シナリオで観測した結果、時間切れ激減（最大 29 → 2）と スイートスポット (p2/d4) 維持を確認。 一方で 守備型 KO 中央値 +20% は Step C 判定システムの導入動機を観測ベースで裏付けた。 Identity Fallback (roundEnabled=false) は Step A baseline と MD5 完全一致。

1. 背景 — Step A から Step B への接続

Step A では roundEnabled=false 既定で従来挙動を bit-identical に保ったまま、 新規定数 (ROUND_LENGTH, NUM_ROUNDS, INTERVAL_LENGTH, STAMINA_INTERVAL_REGEN) と新規 state フィールド (round, roundFrame, matchPhase, roundStats) を整備した。 Step B の目的は、その配管を実際に駆動して 「ラウンド → インターバル → ラウンド」 の試合構造を機械化することにある。

  • 既定 ON 化: roundEnabled を false → true に切替（UI も checked）
  • 境界処理: roundFrame >= ROUND_LENGTH で endRound(s) 発火
  • インターバル: matchPhase="interval" 中はスタミナ +2.0/F 回復、AI 停止、予測 decay 停止
  • 部分リセット: HP / スタミナ / 予測分布は持ち越し、フレーム状態と位置はリセット
  • UI 進行表示: 「R2/3 残り 145F」「休憩中 残り 35F」「試合終了」を roundDisplay に出力

Step A レポートで予告した「endRound / stepInterval を有効化した瞬間に 初めて出力差分が出る — それが Step B によって生じた差だと確定的に切り分けられる」が 本 Step で実現された。Identity Fallback (roundEnabled=false) は MD5 一致を維持。

3. 部分リセット (partialReset) の状態管理

ラウンド境界で 何を持ち越して何をリセットするかが、判定システム導入後の試合構造を決める。 実物ボクシングのコーナー休憩を機械化する方針で以下の通り整理した。

項目扱い理由
hp 持ち越し 実物通り。前R で受けたダメージは次R に引き継ぐ
stamina 持ち越し + インターバル中 +2.0/F 回復 「コーナーで休憩」をメカ化。60F × 2.0 = +120 で実質完全回復
predictedCounts / predictedOpp 持ち越し 実物ボクサーは観測を忘れない。 interval 中は decay も停止（R2 対策）
moveKey / phase / fip / stun / stopRec リセット フレーム単位の状態。インターバルで一旦切れる
decTimer / plan / scores リセット 次R で再判断（前R の決断は持ち越さない）
atkLanded / atkWhiffed リセット フレーム状態
x（位置） INIT_X_P / INIT_X_E に戻す 「コーナーに戻る」を表現

持ち越し / リセットの選定は、「実物の何を機械化するか」「Step A レポート時に R2 として警戒した predictedCounts decay 停止が必要か」の 2 軸で決定。 結論ありきの調整ではなく、設計根拠ベースで固めた。

6. 構造的解釈

時間切れ激減の機械的説明

Step A baseline では 長期戦になると AI / 距離 / 予測分布が定常状態に陥り抜けられない という構造があった。Step B のラウンド境界は次の効果を機械的に与える:

  • 位置を INIT_X_P / INIT_X_E に戻す → 距離関係の固定化が解ける
  • moveKey / phase / fip をリセット → 硬直ループから抜ける
  • decTimer / plan / scores をリセット → 次R で再判断（前R の凝り固まった決断を捨てる）
  • スタミナを実質完全回復 → 「動けないから殴れない」状態を解消

これらは 同時に 発火することで停滞ループから機械的に脱出させる。観測上、 非対称 lag 0/0（最も停滞しやすい設定）で時間切れ 43 → 22 へ約半減、両守備 p3/d5 で 29 → 2 へほぼ消失。

守備型 KO 中央値 +20% — Step C 判定システムの導入動機

守備型 p3/d5 で KO 中央値 615F → 742F (+20%)、 非対称 lag の高 lag 帯でも KO 中央値が +17〜+24% 後退。 「決着するけど遅くなる」方向の変化。

これは構造的に予想される現象: ラウンド境界でリセットされるたびに「もう一度仕切り直し」が 入るため、守備型同士の張り合いでは 累積ダメージ蓄積が間延びする。

ユーザー指摘「ガードが多くなる問題は、ガードが問題ではなく、ガードを崩す機構が無いことが問題」 の 機械的裏付け。Step C で判定システム（攻撃した側がポイントを取る）を導入することで、 「ガード一辺倒 = ポイント負け」という構造的代償が生じ、KO に頼らない決着経路が成立する見込み。

「ガードを崩す機構」B/C（スタミナペナルティ／ガードクラッシュ）の必要性を、 Step C 導入後の観測で再評価できる土台が整った。 理念整合: 「設計目的はあくまで試合構造の機械化、ガードへの代償は副作用」（Step A レポート理念欄）。

10. 教訓

Step A → Step B 接続の流儀（再確認）

段階3 (α) / 段階4 / 段階5 ともに「Step A で配管し Identity を保存、 Step B で既定を ON にして式を駆動する」という同じ流儀を採用している。 本 Step B でも:

  • roundEnabled=false でフォールバック PASS を維持 → 「Step B の差分は全て Step B 由来」と切り分け可能
  • verify.js への ELEM stub 同期を最初から実施（R6 教訓継続適用）
  • verifier 観測ベースの数値判断: KO 中央値 +20% を Step C 動機として明示し、 「結論ありきの調整は禁止」を踏襲

R2 の取り扱い — 「MT=0 では検証不能」を明示する勇気

predictedCounts decay 停止 (interval 中の updatePrediction early return) は MT=0 では 原理的に効果が出ない。「動いているはず」と曖昧に書かず、検証不能と明示して MT=0.3 再測へ送る。これは段階4 Step B の「観測空白活用率 △ 測定不能だが起動証拠あり」と同じ流儀。

守備型 KO 中央値 +20% を「悪化」と書かない

ラウンド境界による仕切り直しで KO が遅れる現象は 構造的に予想される副作用であり、 本 Step の故障ではない。これを Step C の判定システム導入動機として位置づけることで、 プラン (A/B/C) の連続性を観測ベースで担保する。
