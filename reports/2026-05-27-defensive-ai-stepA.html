1. 背景

perception-lag 中間レポート で、 守備 AI に構造的問題 3 点が抽出されている:

  • ① threatened フラグ依存 — ガード score が threatened の有無で 0.10 / 1.20 と段差し、lag 下では threatened が立つ前に被弾
  • ② ガード切替コスト = 0F — 段階 2 時点の瞬時切替前提が lag 導入後も残存
  • ③ 一方的試合 — 両者 lag=5F+ で守備姿勢が出ず、先手 KO 一辺倒

これに対するプラン ai-gleaming-hollerith.md は構造改善を 3 段に分割している:

Step役割挙動への影響
A（今回） 配管 + 観測フック追加 無し（既定 0 / no-op）no-op
B guardScore 式の構造修正（threat 依存比 3.7× → 2.3×）、guardCommitment 既定 0.5 有効化 あり予告
C guardSwapStartup=3F を有効化（FSM レベルの切替コスト） あり予告

Step A の役割は、後段 (B/C) で式を動かしたときに 「どこが何によって変わったか」を機械的に切り分けるための基準 を作ること。 段階 1 Step A、段階 2 Step A と同じ流儀（配管先行 → 値変更）。

2. 実装内容

2-1. src/index.html の 4 配管（すべて既定 0 で no-op）

① PERS 8 軸目 guardCommitment 追加 既定 0

src/index.html l.380-386。ガード方向の継続選好度 ∈ [0, 1]。 既定 0 では従来通り「毎フレーム白紙」の guard 選好になる。 Step B で既定 0.5 を有効化し、ガード方向の継続性を入れる予定。

② MOVES ガード 6 種に guardSwapStartup フィールド 既定 0

src/index.html l.286-289, l.300-301。 別 target （head ↔ body）の guard 系 move に切り替えるときに発生する追加 startup フレーム数。 既定 0 では従来の瞬時切替が温存される。Step C で 3F を有効化する予定。

③ startAction の swap 検出 読まれるが 0 加算

src/index.html l.524-529。 直前 moveKey が guard 系 & 新 k も guard 系 & target 不一致のとき、 ch.stopRec = MOVES[k].guardSwapStartup || 0 を加算する。 現在は全 guard 系 move の guardSwapStartup が 0 のため、加算量は 0F のままで挙動は不変。

④ scoreActions の guardCommitment 倍率 既定 0 で ×1.0

src/index.html l.473-477。 self.guardTarget と一致する guard 行動には ×(1 + 0.5·gc)、 不一致には ×(1 - 0.5·gc)。 self.guardTarget が null のときは両方 ×1.0。 gc=0 （既定）では一致／不一致いずれも倍率 1.0 で従来挙動と完全に同じ。

3. Identity Check 結果

PASS（完全 bit-identical）

Step A の 4 配管はすべて既定値で no-op であり、verify.js での新旧出力比較は 新規ブロック以外完全一致。

  • 単発比較: seed = 1, 2, 3 × (p0/d0)(p2/d4)(p3/d5)(p5/d8) の フレーム数・KO・攻撃内訳・ヒット内訳がすべて完全一致
  • 50 戦集計: 旧 4 段すべての P 勝 / E 勝 / 時間切れ / 平均 F / KO 中央値 / スタミナ / 総攻撃 / ヒット内訳 / 技 kind outcome / whiff 内訳 / whiff 原因 が完全一致
  • 差分: 新規出力ブロックのみ （新 lag 3 段 [4,6][5,8][7,10] と性格 3 ケース、 --- 段階3 (α) Step A 新指標 --- セクション）

Identity Check は「式に手を入れる前に配管を済ませる」スタイルの定型確認。 これにより Step B/C で値を動かしたときの差分が、 「Step B で動いた」「Step C で動いた」と切り分けて読める。

4. 新指標フックの出力サンプル

verify.js に追加した 2 指標（ガード切替頻度 / 殴り合い率）は読み出されているが、 Step A では構造変更が無いため値そのものは「既定状態でのベースライン」を示すに留まる。

シナリオ ガード切替 / 試合 合計（50 戦） 殴り合い率
非対称 baseP / baseE (lag=0/0) 13.0 回 649 9.0%
両攻撃型 (lag=3/5) 0.3 回 16 28.7%
両攻撃型 (lag=5/8) 0.2 回 8 28.3%
両守備型 (lag=3/5) ジャブ被弾率 79% / フェイント試行 569 など、攻撃型と劇的に違うプロファイル（数値詳細は verify 出力を参照）

読み取り

  • ガード切替頻度: lag が増えると「現 target の guard 自体を選ばなくなる」ので切替回数も下がる （0.3 → 0.2）。lag=0 の理想条件下でも 13 回/試合と多く、「白紙の再決定」がそのまま切替に出ている
  • 殴り合い率: lag=0 で 9% に対し lag=3/5 以上は 28% 前後と跳ねる。これが「一方的試合」の数値裏付け
  • いずれも 「フックは読まれているが、現状は何の補正もかからない（既定 no-op）」 状態の確認。 Step B / C で値を動かしたときに、この同じ指標がどう変位するかが評価軸になる
