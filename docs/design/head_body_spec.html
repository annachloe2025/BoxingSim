仕様: 頭/腹の打ち分け — 段階2

駆け引きを生む3段階ロードマップの第2段階。攻め手 2 ×ガード 2 のマッチング機構を機械的に追加する。
段階2 単独では「読み合い」は立ち上がらない（完全情報＋無遅延では守備が常に追従する想定）。 読み合いとして立ち上がるのは段階3（反応遅延）と組んだとき。ここでは土台を機械的に積む。

方針メモ: 機械的仕組みを先に積み、面白さは結果として立ち上がる ／ 関連: FSM 棚卸し（B2 防御の細分化 / 攻撃の細分化） ／ 段階1 スタミナ仕様 ／ TASKS.md

0. スコープ確定（2026-05-26 相談）

  • 段階2 = 頭/腹の打ち分け のみ。フェイント は本段階から外し、後段に分離。
  • 理由: フェイントは「相手の能動的な待ち状態（カウンター待ち・読み構え）を誤発火させる」道具。 誘発対象となる「待ち状態」が今の FSM には存在しない。先に作っても損な選択肢にしかならない。
  • フェイントは「待ち状態」（カウンター予約・反応構え等）の設計とセットで後段に。段階3（反応遅延）の後で組み合わせるのが自然。

段階2 完成後に立ち上がる線（観察対象として事前記録）: 頭/腹2択を入れると、守備側はバックステップで両択回避が最適化される → 結果として コーナーに追い詰められる事態が増える → バックステップを回数制限で抑える、という流れが 正常な駆け引きとして発生するはず。段階2 単独で対策せず、コーナー機構（fsm_gap_inventory D1）と セットで解く。

1. 基本コンセプト

  • 攻撃に target ∈ {head, body} を持たせる。攻撃側は head/body のどちらを狙うかを毎回選ぶ。
  • ガードにも target ∈ {head, body} を持たせる。守備側は head/body のどちらを守るかを毎回選ぶ。
  • 判定: 攻撃 target == ガード target → ブロック成功（ダメージ0）／ ミスマッチ → ノーガード相当（フルダメージ）。
  • 段階2 では target は完全公開（観測に出す、★印付き）。曖昧化は段階3 で行う。
  • 性格軸は追加しない（既存5軸のまま）。target 選択は「相手のガード逆」一本で機械的に決める。

3. 判定ロジック

既存の「接触検出 → 効果適用」の2パス方式は維持。検出パスで攻撃と被弾者状態を記録、適用パスで以下のロジックを通す。

// 検出パスで記録した接触ごとに、適用パスで:
A = 攻撃の target （'head' | 'body'）
G = 被弾者の現在の guardTarget （ガード/ガード移動中なら 'head' | 'body'、その他は null）

if (G === null) → ノーガード判定（既存と同じ damage 計算、のけぞり付与）
else if (G === A) → ブロック成功（damage = 0、ガード硬直付与）
else → ノーガード相当（既存の damage 計算、のけぞり付与）
  • カウンター倍率（×1.5・始動つぶし）／相打ち／パニッシュの既存ロジックはそのまま流用。target は「ブロック成立か否か」の判定にのみ使う。
  • guardTarget はキャラの派生プロパティ。moveKey が guard-* / guard-fwd-* / guard-back-* のいずれかなら、その末尾から導出する（別フィールドは持たない）。

4. 表示と観測

画面表示

  • キャラカード: actionLabel に上下記号を付加。 ▲ 頭ジャブ / ▼ 腹ジャブ / ガード▲ / ガード▼ など。 target を持たない行動（前進・後退・待機）は記号なし。
  • 間合いゲージ（リーチ帯）: 既存の自分=緑 / 敵=赤、発生中=濃 / 始動中=淡 の表示はそのまま。 帯の上に小さく ▲ / ▼ を重ねる（色分けはせず記号のみで判別）。
  • 行動候補テーブル（score 表示）: 攻撃が 4 行 / ガードが 6 行に増えるので、target 列を別カラムに切り出し、種別ごとにソートして見やすく。

観測スキーマ（buildObs）

self（完全把握）

  • action: 'jab/head' | 'straight/head' | 'body-jab/body' | ... | 'guard-fwd/body'
  • guardTarget: 'head' | 'body' | null （ガード系状態でないときは null）
  • 他の既存フィールドは変更なし

opponent（★ 段階3で曖昧化対象）

  • action: 同上の形式で完全公開（★）
  • guardTarget: 完全公開（★）
  • 段階3 で「active F の直前まで target を隠す」「反応遅延 NF」等で曖昧化

relative ブロック

  • reachableAttacks: 自分が今出せる攻撃のうち射程内のもの一覧 ['jab','straight','body-jab','body-straight'] から該当を返す。 target 込みの最適評価のため。既存の inRangeForJab 等はそのまま残す。

技編集テーブル

  • 行が 10 行＋移動類 5 行で計 15 行になるので、kind 列でソート（attack 4 → guard 6 → move 4 → idle）。
  • 新列 target（head/body/—）を追加（リードオンリー、MOVES の定数）。

5. AI への統合

新しい性格軸は追加しない（既存 ag/df/pd/pa/sa のまま）。scoreActions に 二要素の倍率 を掛ける形で「相手のガードを見て逆を狙う／相手の攻撃に合わせる」を機械的に表現。

攻撃側（4 攻撃の score 倍率）

oppGT = opponent.guardTarget （null / 'head' / 'body'）

if (oppGT === null) → head-* も body-* も等倍（既存ロジックそのまま）
else if (oppGT === attackTarget) → score × 0.3（ガード方向に突っ込む、避けたい）
else → score × 1.5（逆を突く、選好）

意図: 相手が頭を守っていれば腹を、腹を守っていれば頭を狙う。両者完全公開なので守備側も即追従 → 段階2 単独では「攻撃が通らない / 時間切れ多発」になる想定。それを観測して段階3 の必要性を裏付ける。

守備側（6 ガードの target 選択）

oppAct = opponent.action （phase + target）

if (oppAct が攻撃 startup 中) → そのattackTargetと同じ target のガード系 score × 2.0、逆 target × 0.5
else if (oppAct が攻撃 active 中) → 判定確定済み、target 据え置き（変更しても無意味）
else (非攻撃状態) → head をデフォルト選好（×1.2）

ガード移動 vs その場ガード の選び方は既存ロジック（pd 防御性 と pr 好む間合い）を流用、target だけ上記で決める。 pd が高いほどガード系全般を選びやすく、その中で target は上の倍率で決まる、という二段構え。

guardTarget の連続性

  • 1 回 startAction で選んだ target はその行動が続く限り保持。DECIDE_EVERY=3 のタイミングで再評価され、target を変える場合はガードを再 startAction（即時切り替え）。
  • 最小実装では切り替えコストなし。verify.js で「切り替え多発の異常」が出たら、次イテレーションで 1F の idle 強制等を入れる。

性格との関係

  • 既存 ag（攻撃性）/ df（防御性）はそのまま、攻撃/防御の総量を制御。
  • 頭/腹の選好は性格化せず、「相手のガード逆」一本（土台方針）。後で必要なら 6 軸目 bodyPreference を追加できる構造。

6. 実装の進め方（3 ステップ分割）

スタミナと同じく、機械的に1段ずつ進め、各段で verify.js で挙動を観察してから次へ。

Step A

データと判定の刷新

  • MOVES に target 付与・新規 7 行追加・既存 3 行リネーム（jab→jab[head], guard→guard-head 等）
  • 接触判定に target マッチ照合を追加（既存の防御判定を target 比較に置き換え）
  • 観測 self/opponent.action / guardTarget を新形式に
  • キャラカード・リーチ帯・技編集テーブルを新形式に
  • AI は既存ロジックのまま（target 倍率はまだ入れず、head 固定で動かす）
Identity Check: 全部 head のままで以前と同じ挙動（seed 1/2/3 の試合長・勝者が前回と一致）になることを確認。
Step B

AI 攻撃 target 選択

  • scoreActions に「相手 guardTarget の逆を狙う」倍率を追加（×1.5 / ×0.3）
  • 攻撃側だけが target を選ぶ。守備側はまだ head 固定
  • relative.reachableAttacks 等の利用
観察ポイント: 50戦して攻撃 head/body 比率と「ガード貫通率」を計測。守備が head 固定の状態で body 攻撃が通ることを確認。
Step C

AI 守備 target 選択

  • 守備側の「opponent.attackTarget に合わせる」倍率を追加（×2.0 / ×0.5）
  • 非攻撃時の head デフォルト選好（×1.2）
  • 性格は既存5軸そのまま
観察ポイント: 完全公開＋無遅延では守備が常に追従し攻撃が通らない、を観測。時間切れ・ガス欠が増える想定。それが段階3 の動機になる。

7. 検証計画（verify.js で測るもの）

Step A 後（AI 既存ロジック）

  • Identity Check: seed 1/2/3 の試合結果が段階1完了時点と完全一致。target 機構の追加で挙動が変わっていないこと。

Step B 後（攻撃側だけ target 選択）

  • 50戦して: 攻撃の head/body 比率（攻撃側が「逆」を選んでいる比率）
  • 「守備 head 固定 vs 攻撃 body」のヒット率（守備が頭固定なので body が通るはず）
  • KO 数 / KO 中央値 F の変化

Step C 後（守備側も target 選択 = 段階2 完成）

  • 時間切れ率（Step B 後と比較。守備有利で増える予想）
  • KO 数 / KO 中央値 F の変化
  • 攻撃 head/body 比率（攻撃側の偏りが出るか）
  • 「ガード切り替え遅れヒット」発生回数（無遅延なので 0 のはず）
  • 平均スタミナ・スタ0 F 合計（時間切れ多発でガス欠が増えるはず）
  • guardTarget の切り替え頻度（ジッタ発生有無）

ここで観察するのは「狙ったか」ではなく「結果として何が立ち上がったか」。 想定通り「守備有利で時間切れ多発」が出れば、段階3（反応遅延）に進む動機が立つ。 想定外（攻撃 head/body 極端な偏り、ガード切り替え無限ループ、特定セットで KO 量産 等）が出たら、それも観察対象として LOG.md に記録。
