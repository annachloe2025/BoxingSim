仕様: 技データベースの参照層（Step 1a）

目的（1行）: 技を放つために既にある技データ＋現在の状況を、AI（キャラ）が まとめて参照できる surface（読み取り専用 API）を作る。挙動は一切変えない（bit-identical）。

分類: 参照層の追加のみ。点数計算（scoreActions）には未接続。 これは段階1b（参照層を実際に意思決定で使う）への足場であり、それ自体は AI の選択を変えない。 対象実装: src/index.html。

1. 既にあるもの／新規に作るもの の切り分け

冗長な足場を作らないため、まず現状コードを精査した。 結論: 生データはほぼ全部既にある。本当に新規なのは「1 つの move key からフレームデータを引く薄い getter」と 「それらを 1 オブジェクトにまとめて返す situation 関数」の2点だけ。

参照したいもの区分現状の所在 / 備考
各技の startup / active / recovery / reach / damage / blockstun / hitstun 既存（生データ） MOVES[key]（317–339行）に全フィールドが既にある。ただし「key から構造化して引く関数」は無い。
各技の kind / target / chipRatio / staminaCost 既存（生データ） 同上。MOVES[key] に格納済み。
自分・相手の現在フェーズ（state） 既存 stateLabel(ch)（733行）がフル粒度（待機/移動/始動/発生/硬直/のけぞり/ガード）を返す。
観測では self.state / opponent.state（★detailLag 遅れ）。
そのフェーズの残りF（phaseRemaining） 既存 phaseRemain(ch)（1166行）。観測の self.phaseRemaining / opponent.phaseRemaining（★）。
相手が次に動ける/ガードできるまでのF 既存（流用） busyFrames(ch)（757行）が「あと何F行動不能か」を導出済み。観測の self.busyFrames / opponent.busyFrames。
「ガードできるまで」も同じ値（行動可能になった瞬間にガードも選べる ＝ canActNow 成立）。新規導出は不要、別名で見せるだけ。
敵との距離（間合い） 既存 relative.distance（1264行）。射程内判定は relative.inMyReach[key] / relative.inOppReach。
move key → フレームデータの構造化 getter 新規（薄い） MOVES[key] から欲しいフィールドだけを正規化して返す関数。これだけが新規本体。
現況を1つにまとめた situation オブジェクト 新規（集約） 上の既存値（state/phaseRemaining/busyFrames/distance 等）を1関数で集約。新しい計算は無く、既存値の取りまとめのみ。

切り分け要約: 生データ・導出値はすべて MOVES と buildObs/busyFrames に既にある。 新規に作るのは「key で技データを引く getter」と「現況集約 situation」の2つの薄いラッパだけ。 既存の buildObs や busyFrames は再実装しない（呼ぶだけ）。

2. 提案する参照 API（読み取り専用・副作用なし）

2-1. moveData(key) — 1 つの技のフレームデータを引く 新規

入力: key （MOVES のキー文字列。例 "ジャブ" / "強ストレート上"） 戻り値: { ...フィールド... } 、未知 key なら null

戻り値フィールド一覧:

フィールド意味 / 出所
key引数の key をそのまま返す（呼び出し側の照合用）
kind"idle" / "attack" / "move"（MOVES[key].kind）
target"head" / "body" / null（攻撃・ガードのみ）
startup始動F（attack のみ。素の値。0 は active から始まる技）
active発生F（当たり判定アクティブ）
recovery硬直F
reachリーチ（距離単位。move/idle は 0/undefined → 0 正規化）
damageヒット時ダメージ
hitstunのけぞり硬直F（被弾側に入る）
blockstunガード硬直F
chipRatioガード時の削り比率（通常 0、強ストレート 0.3）
staminaCost消費スタミナ（攻撃は1回、移動は per-F）

素の値を返す（スタミナによる startup 延長 effectivePhaseLen は適用しない）。 理由: ここは「技スペックの台帳」であり、現在状況込みの実効値は situation 側 or 1b の責務。台帳と現況を混ぜない。
補助として allMoveKeys()（= MOVE_ORDER を返す薄い getter、または MOVE_ORDER をそのまま参照）で全 key を列挙可能にしておく。

2-2. situation(obs) — 現況を1オブジェクトに集約 新規

入力: obs （既存 buildObs(...) の戻り値。新規に観測は作らない） 戻り値: { self, opponent, between }

戻り値フィールド一覧（すべて obs から取りまとめるだけ。新規計算なし）:

フィールド意味 / 出所
self.state自分の現フェーズ（obs.self.state）
self.phaseRemaining現フェーズ残りF（obs.self.phaseRemaining）
self.framesUntilActable自分が動ける/ガードできるまでのF（obs.self.busyFrames の別名）
self.guardTarget今守っている部位 head/body/null（obs.self.guardTarget）
self.attackTarget今出している攻撃の部位（obs.self.attackTarget）
opponent.state相手の現フェーズ（obs.opponent.state。★detailLag 遅れの観測値そのまま）
opponent.phaseRemaining相手フェーズ残りF（obs.opponent.phaseRemaining、★）
opponent.framesUntilActable相手が動ける/ガードできるまでのF（obs.opponent.busyFrames の別名）
opponent.isAttacking相手が攻撃中か（粗知覚 presenceLag、obs.opponent.isAttacking）
opponent.isGuarding相手がガード中か（obs.opponent.isGuarding）
opponent.guardTarget相手のガード部位（obs.opponent.guardTarget、★）
opponent.attackTarget相手の攻撃部位（obs.opponent.attackTarget、★）
between.distance間合い（obs.relative.distance）
between.closing接近中か true/false/null（obs.relative.closing）
between.inMyReach各攻撃 key が届くか {key:bool}（obs.relative.inMyReach）
between.inOppReach相手の最大リーチ内にいるか（obs.relative.inOppReach）
between.frameAdvantageフレーム有利（正=自分が先に動ける、obs.relative.frameAdvantage）

知覚遅延の扱いは観測に従う: opponent.* は ★ の遅延付き値をそのまま中継する（新たに生値を覗かない＝知覚モデルを壊さない）。 「動ける/ガードできるまでのF」は別計算しない。busyFrames が既に同じ意味で、行動可能になった瞬間にガードも選べる（canActNow）ため同一値で正しい。

3. どこから呼べるようにするか

  • 両関数は src/index.html のグローバルスコープに置く（既存 MOVES / buildObs / busyFrames と同じ層）。AI の意思決定文脈（decideUtility → scoreActions 周辺）から呼べる位置。
  • moveData は MOVES を読むだけ。situation は既存 buildObs(...) の戻り値を引数に取る（観測を二重生成しない）。
  • 呼び出しは追加しない。scoreActions 含め既存の意思決定経路からは呼ばない（呼べば挙動が変わる）。定義を置くだけ。実際に使うのは段階1b。
  • 任意（推奨）: 観測表示パネル（renderObs 付近）に situation のダンプを表示だけ追加するのは可。ただし表示は frames/数値計算に影響しないこと（verify.js は DOM 描画を回さないので Identity に無関係だが、念のため副作用ゼロを守る）。

4. 受け入れ条件

Identity Check PASS（最優先・必須）
参照層（moveData / situation / allMoveKeys）を追加するが、点数計算には未接続。 したがって tools/verify.js の出力が編集前後で MD5 完全一致すること。 （直近 baseline は LOG.md 末尾の MD5。実装者は編集前に baseline を取得し、編集後に再取得して照合する。）
  • moveData(key) は MOVES の全 key で 2-1 の全フィールドを返す。未知 key で null。
  • situation(obs) は buildObs の戻り値から 2-2 の全フィールドを取りまとめて返す（新規計算なし＝obs と内容が一致）。
  • 両関数とも副作用ゼロ（引数や MOVES/frames を変更しない、Math.random 不使用）。
  • 既存関数（buildObs / busyFrames / scoreActions 等）は未編集。

5. Out of Scope（今回は絶対にやらない・1b 以降の予定）

  • scoreActions への接続・挙動変更 → 段階1b
  • 対策・カウンターの導出や手書き（どの技にどう返すか）→ 1b 以降、計算機で導出予定
  • 予測（0F）・スキル・キャラ別所有・戦略 → すべて後段。今回は触れない
  • 実効値の合成（スタミナ込み startup 等を situation に混ぜること）→ 台帳と現況は分離。1b で必要なら導出

今回の Step 1a は「既にある技データ＋現況を、AI がまとめて読める形にするだけ」。 読めるようにするのが目的で、読んだ結果で何を選ぶかは 1b の責務。よって挙動は bit-identical。
